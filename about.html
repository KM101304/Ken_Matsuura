About

How I got here and how I think about building things.

Background

I did not come from a traditional computer science background. No algorithms course, no university CS lab, no textbook progression from binary to data structures to distributed systems.

I came through business, operations, and the pressure of needing to make ideas real. The path was nonlinear and entirely driven by one question: how does this actually work?

How I learned

My first real breakthrough came when I understood that software systems are not magic. They are interconnected files, APIs, processes, databases, interfaces, workflows, and infrastructure. Once I understood that structure, software stopped looking like an impossible wall and started looking like a system I could navigate.

I learned by doing. I opened VS Code, cloned repositories, ran local servers, broke dependencies, misconfigured environments, fought authentication flows, debugged API errors, rebuilt frontends, rewired backends. Debugging became the classroom. Every error message was a lesson in how systems actually behave under real conditions — not how documentation says they should behave, but what they do when pieces don't align.

AI accelerated the process but did not remove the hard part. It moved the hard part. The hard part became knowing what to build, understanding how systems connect, detecting bad architecture, debugging what actually failed, and explaining requirements precisely enough to execute. That skillset is harder than memorizing syntax. It is also the skillset that matters.

How I think about software

My background in business operations shaped how I approach building. I was never interested in code as an isolated technical exercise. I cared about leverage, workflows, monetization, user behavior, bottlenecks, automation, and execution. I began seeing software as infrastructure — not a product category, but a substrate for solving operational and human problems.

A system is not good because it is technically elegant. It is good because it solves the actual problem it was built to solve, reliably, without collapsing under real conditions.

The early projects were messy. Some broke constantly. Some had bad architecture. Some were rebuilt from scratch. But every broken project taught something concrete. I learned by compressing massive amounts of experimentation into short periods and shipping immediately rather than waiting for the plan to be perfect.

What I use

Frontend: React, Next.js, Tailwind, plain HTML and CSS.
Backend: Python, Flask, Node.js.

Payments: Stripe.
Automation: n8n, scraping pipelines, scheduled jobs.
